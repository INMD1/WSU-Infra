# 2026-05-04 쿼터 시스템 수정 작업

## 📋 작업 개요

사용자가 로그인할 때 자동으로 **초기 쿼터 (할당량)** 가 생성되도록 수정했습니다.

---

## 🎯 문제 상황

### 기존 문제
1. 사용자가 로그인해도 **쿼터 레코드가 생성되지 않음**
2. 프론트엔드가 하드코딩된 값을 표시
3. `/api/quotas?userId=<owner_id>` API 가 userId 파라미터를 받지 못함

### 예시
```
기존: 프론트에서 { vCPU: 3, RAM: 4GB } 하드코딩 표시
수정: 실제 DB 에서 쿼터 값을 가져와서 표시
```

```

   ↓
2. POST /api/auth/login
   ↓
3. 백엔드: 유저의 쿼터 레코드 확인 → 없으면 생성
   ↓

   ↓
5. 프론트: localStorage.setItem('owner_id', owner_id)
   ↓
6. Dashboard 로딩
   ↓
7. GET /api/quotas?userId=<owner_id>
   ↓
8. Response: { quota, usage, remaining }
   ↓
9. 쿼터 사용량 표시
```

## 💡 참고사항

### 기존 유저의 쿼터 업데이트

이미 생성된 유저의 쿼터 레코드는 **수동으로 업데이트**해야 합니다:

```sql
-- 모든 유저의 쿼터를 새로운 기본값으로 재설정
UPDATE quotas SET 
  max_vcpu_total = 3,
  max_ram_gb_total = 4,
  max_disk_gb_total = 100;
```

또는 **Drizzle Studio**에서 `quotas` 테이블을 직접 수정할 수 있습니다.

### 왜 로그인 시에 쿼터를 생성할까?

1. **사용자 경험**: 로그인하면 바로 쿼터 정보를 볼 수 있음
2. **일관성**: 모든 유저에게 동일한 초기 쿼터 제공
3. **간단함**: 별도의 초기화 작업 없이 자동 처리

## 📝 기술적 설명 (초급자를 위한)

### `owner_id` 란?

- 각 사용자를 고유하게 식별하는 **UUID** (Unique Identifier)
- 예: `08892577-19a7-44cc-905a-97ec66f8a4d3`
- 사용자의 VM, 쿼터 등을 연결하는 **키** 역할

### 쿼터 (Quota) 란?

- 사용자가 사용할 수 있는 **리소스 할당량**
- 예: "vCPU 3 개까지 사용 가능", "RAM 4GB 까지 사용 가능"
- 리소스 할당량을 초과하면 VM 생성이 불가능

### `??` 연산자 (Nullish Coalescing)

```typescript
quotas.quota?.max_vcpu_total ?? 3
```

- 왼쪽 값이 `null` 또는 `undefined` 이면 오른쪽 값 반환
- 예: `quotas.quota?.max_vcpu_total` 가 없으면 `3` 반환
